HOM

HOM — Harness Orchestration Management

A Rust TUI terminal multiplexer that spawns 7 AI coding agent CLIs in visual panes. Each pane runs a real terminal emulator (ghostty default, vt100 opt-in fallback), mapped cell-by-cell into ratatui buffers. The orchestrator coordinates inputs, outputs, and YAML-driven DAG workflows between harnesses.

10-Crate Workspace

Strict dependency rules: hom-core is the root with zero internal deps. Each layer depends only on hom-core (and hom-plugin for adapters). Only hom-tui and main.rs pull everything together.

Data Flow — The TUI-inside-TUI Pipeline

Every byte from a harness flows through this pipeline. A bug anywhere means garbled output.

Mouse Passthrough

Mouse events are captured via EnableMouseCapture (crossterm) at startup and released via DisableMouseCapture on quit. Non-focus mouse events (click, drag, scroll) that land on the focused pane are encoded as X10 protocol bytes and written to the PTY stdin so the harness receives them natively:

X10 Encoding

ESC [ M <Cb> <Cx> <Cy>
  • 3-byte payload after ESC [ M
  • Cb = button bits + modifiers
  • Cx / Cy = column + row (1-based, +32 bias)

Event Routing

InputRouter dispatches mouse events
  • Focus mouse events handled by HOM (pane focus, resize handles)
  • Non-focus events forwarded to focused PTY as X10 bytes
  • Harness receives mouse events as if running in a real terminal

Lifecycle

crossterm terminal state
  • EnableMouseCapture in main.rs setup
  • DisableMouseCapture in App::shutdown() teardown
  • Restored on Ctrl-Q, :quit, and process crash

Workflow Engine — YAML DAG Execution

Workflows define steps with dependencies. The engine builds a DAG (petgraph), does topological sort, and executes steps across harnesses with templating, retries, conditions, and SQLite checkpointing.
